Same code. Five pilots.

def get_user(id): query = "SELECT * FROM users WHERE id = " + str(id) return db.execute(query) # what could go wrong
grumpy-senior
SQL concatenation in 2024. Not parameterized queries — concatenation. Also, id shadows a Python builtin. Fix it. Both of them. Today.
hype-founder
Love the velocity. Small note: this will absolutely get us hacked, which is bad for the Series B. Parameterized queries, five minutes, let's go.
paranoid-security
Have you considered SQL injection? Because I have. At length. str(id) does zero validation. A user passes 1 OR 1=1 and you've handed them your entire database. Parameterize this immediately.
academic
This exhibits a classic injection vulnerability characterized by Halfond et al. (2006). Parameterized queries have been consensus best practice for two decades. The corrected form is O(1) in query complexity.
zen-master
The query and the input are not separate things. You have made them one. This is the source of your problem. Separate them. Then sit with why you merged them in the first place.

How it works

1
Install the plugin
One command. Available across all your projects.
2
Pick a pilot
Run /snarkpilot:set-pilot grumpy-senior inside Claude Code, or use /config → Output style.
3
Work normally
Ask for code reviews, debugging help, architecture advice — same correct answers, wildly different energy.
4
Auto-load per project (optional)
Drop a .pilot file in any repo. The persona loads automatically every session.
5
PR reviews via GitHub Actions (optional)
Add the workflow to any repo. Every PR gets reviewed in character and posted as a comment.
